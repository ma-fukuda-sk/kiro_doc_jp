# Concepts（コンセプト）

Kiroの構造化された仕様システムは、プロダクト要件と技術実装を橋渡しします。このシステムは3つの主要ファイルを生成します：

## 生成されるファイル

1. **requirements.md** - 構造化されたEARS記法でユーザーストーリーと受け入れ基準を記録
2. **design.md** - 技術アーキテクチャと実装上の考慮事項をドキュメント化
3. **tasks.md** - 追跡可能なタスクを含む詳細な実装計画を提供

## ワークフローフェーズ

仕様プロセスは4つの順次ステージに従います：

### 1. Requirements Phase（要件フェーズ）

EARS（Easy Approach to Requirements Syntax）記法を使用してユーザーストーリーを定義します。

**EARS記法フォーマット:**

```
WHEN [条件/イベント] THE SYSTEM SHALL [期待される動作]
```

**例:**

```
WHEN a user submits a form with invalid data THE SYSTEM SHALL display validation errors next to relevant fields
```

### 2. Design Phase（設計フェーズ）

技術アーキテクチャ、シーケンス図、システムインタラクションをドキュメント化します。

design.mdファイルには以下が含まれます：
- アーキテクチャ
- データフロー
- インターフェース
- データモデル
- エラーハンドリング
- テスト戦略

### 3. Implementation Planning（実装計画）

作業を明確な成果を持つ個別の追跡可能なタスクに分解します。

### 4. Execution Phase（実行フェーズ）

タスクの完了に伴い進捗を追跡し、必要に応じて仕様を改善します。

## 構造化された要件の利点

- **明確性**: 曖昧さがなく、理解しやすい要件
- **テスト可能性**: テストケースへの直接変換が可能
- **追跡可能性**: 実装を通じて要件を追跡
- **完全性**: 徹底した条件分析を促進

## 実装追跡

tasks.mdファイルはリアルタイムのステータス更新を持つタスク実行インターフェースを提供し、チームが開発進捗を追跡し、現在の実装ビューを維持できるようにします。
